SAGA Pattern 
 choreography
 Orchestrator

It is like a dance 
In choreography , there is no central coordinator , each service publishes event and reacts to event from other services 
Order service -> create order -> OrderCreated -> 
Payment services -> OrderCreated -> publishes payment completed
Inventory service -> listens to payment completed -> publishes stockreserved 

If any sequence breaks , it publishes compensating commands 

Orcehstration 
In orchestration , a central orchestration service controls the workflow 
Saga Orchestrator -> calls order service , create order -> calls payment service , charge card -> calls inventory service , reserve stock 

choreography , few microservices , less complex scenarios , event driven architecture 
orchestration , complex workflows , needs centralized control , need monitoring and retry logic 

For simple event driven architecture , i would go with choreography , where as for complex scenarios , we will be going with the orchestration pattern 
-----------------
CQRS design pattern 
Command and query segregation 

product query service :  /GET  - FetchProducts
product command service : /POST , /PUT ,/DELETE  

example flipkar sale , during the sale days - searches are more and creates are less 

separates writes from reads , independent scaling , mostly used for ecommerce 
read side can have materialized views 
data are synchronized based on teh event , write produces an event which is consumed by read
write can be relational while reads can be elastic 

-----------------
Transaction outbox pattern 

Solves the problem , when the data is to be written to the database and also published for the consumer to pick up the same message . Like data is committed to DB but publishing fail 
In this pattern , data is written to the related table and also to the outbox table , and the data is synched by a separate background job , which picks up the data from the outbox table , and changes the sync status 

-----------------

Event sourcing design pattern 

Instead of storing the final state , it stores the event , and final state is derived by deriving by replaying the events , it is mostly used for high audit API's

Order created , confirmed , shipped n delivered 
also like bankin , 200 deposited , 100 withdrawn , 300 deposited etc...

better to use no-sql for scaling , they can also be stored in message broker like Kafka , event hub , Rabbit MQ etc
----------------
Aggregator pattern 

Aggregator service - instead of the UI calling multiple services , and waiting , it can call an aggregator service , 
The aggregator service in turn will call the services and aggregates information for UI consumption 

Benefits , simplified UI , optimized performance (parallel api calls) , better resilience , security enhancements

Backend for Frontend (BFF)

It is a custom API that sits in between , fronte end App and back end microservices . Instead of front end calling all the different microservices , it  makes a single call to the BFF service , the BFF structures and formats the response expected by the front end .

Circuit Breaker Pattern 

To prevent cascading failure , this particular pattern is implemented , with out which , if the payment services is being called , and it does not give any response , without circuit breaker pattern , system would crash , with circuit breaker pattern , it would block the request , resulting in the system continue to work.
